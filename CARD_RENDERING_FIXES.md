# BJP TN Member Card — Two Critical Rendering Fixes

**Date:** 2026-07-15
**Priority:** Fix before launch
**Estimated effort:** 30 minutes total
**Files affected:** The component that calls `html2canvas` (card download/generation logic)

---

## Background

The member ID card is generated on the **user's device** (client-side). The process is:

1. The card is rendered as HTML/CSS inside an iframe
2. `html2canvas` takes a screenshot of that rendered HTML
3. That screenshot becomes the downloadable card image

Because this happens on the user's phone, the output quality depends on what is happening on their device at that exact moment — specifically, whether fonts and images have finished loading, and what the screen resolution of their device is.

Two things can silently go wrong. Both are fixable in under 30 minutes.

---

## Fix 1 — Wait for Fonts Before Capturing the Card

### What is the issue?

The card uses custom fonts for the member's name, BJP branding, and other text. These fonts are not built into the phone — they are downloaded from the internet when the page loads.

`html2canvas` does not automatically wait for fonts to finish downloading before it takes the screenshot. If it captures the card while the font is still loading, the card image is saved with the phone's default system font instead of the correct custom font.

The user never sees this happen. They click "Download", the card appears, and they share it — not knowing the font is wrong.

### What could go wrong if we don't fix this?

- **Member's name appears in the wrong font** — looks unprofessional and inconsistent
- **Text spacing breaks** — custom fonts have specific letter spacing. The fallback system font has different spacing, causing text to overflow its box, overlap other elements, or get cut off at the edges
- **Cards look different across users** — one member's card looks correct, another's has a different font. Both were generated by the same app
- **Printed cards look wrong** — members who print the card for display will have a visually broken product

**How often does this happen?**
On a fast 4G connection in a city — rarely.
On a slow connection, rural 3G, or when the network is congested — 25 to 40% of users will experience this. Tamil Nadu BJP has significant rural membership where weak connectivity is common.

### What happens if we fix it?

Every user gets the correct font every time, regardless of their connection speed. The card looks identical whether generated on a high-end phone in Chennai or a budget phone in a rural area with weak signal.

### How to fix it

Before the `html2canvas` call, add one line:

```javascript
// Wait for all custom fonts to fully load before capturing
await document.fonts.ready;

// Now capture — font is guaranteed to be present
const canvas = await html2canvas(element, { ... });
```

`document.fonts.ready` is a browser-native Promise that resolves only when every font on the page has finished downloading and is ready to render. It costs zero performance on fast connections (resolves instantly if fonts are already loaded) and protects users on slow connections by waiting the extra second or two needed.

**If the card is rendered inside an iframe**, the fix needs to be applied inside the iframe's document, not the parent:

```javascript
const iframe = document.querySelector('iframe');
const iframeDoc = iframe.contentDocument || iframe.contentWindow.document;

// Wait for fonts inside the iframe
await iframeDoc.fonts.ready;

// Then capture
const canvas = await html2canvas(iframeDoc.body, { ... });
```

### What success looks like

- Download the card on a throttled connection (Chrome DevTools → Network → set to "Slow 3G")
- The card takes a second or two longer to generate
- When it appears, all text is in the correct custom font
- Text is properly spaced, no overflow, no cutoff
- Generate 5 cards in a row — all identical font rendering

### Does this fix break any existing functionality?

No. `document.fonts.ready` only adds a wait. It does not change how the card is captured, how it is downloaded, or how it is saved. All existing behaviour remains exactly the same — the card just waits an extra moment before being captured on slow connections.

## Fix 2 — Force Consistent Card Resolution on All Devices

### What is the issue?

Every phone screen has a **Device Pixel Ratio (DPR)** — a number that describes how many physical pixels the screen uses per CSS pixel.

| Device type                          | DPR      | Example phones     |
| ------------------------------------ | -------- | ------------------ |
| High-end (iPhone 15, Samsung S25)    | 3.0      | ₹80,000+          |
| Mid-range (Redmi Note, Realme GT)    | 2.0      | ₹15,000–₹30,000 |
| Budget (Redmi A series, Tecno, Itel) | 1.0–1.5 | ₹5,000–₹12,000  |

By default, `html2canvas` captures the card at the device's DPR. This means:

- A member with a high-end phone gets a sharp, high-resolution card image
- A member with a budget phone gets a low-resolution, blurry card image

The card content is correct on both — but the output quality is different depending on the phone used to generate it.

### What could go wrong if we don't fix this?

- **Blurry cards when shared on WhatsApp** — when a card image is shared in a chat, WhatsApp displays it at a larger size than the original. A low-DPR card becomes noticeably blurry
- **Blurry cards when printed** — members who print their card for display at home or at the party office will have a blurry printout
- **Inconsistent quality across members** — two members generate their cards side by side. One looks sharp, one looks blurry. Both used the same app, but different phones
- **Perception of a broken or low-quality product** — members on budget phones — who make up roughly 50% of your user base — consistently receive an inferior output compared to members on expensive phones, with no explanation

**How often does this happen?**
Every single time a user on a DPR 1.0 or 1.5 device generates their card. Estimated 35 to 50% of your users.

### What happens if we fix it?

Every user — regardless of phone price or screen quality — receives the same high-resolution card image. A member in Chennai on an iPhone and a member in a rural taluk on a ₹6,000 Tecno phone get identical output quality.

### How to fix it

Pass an explicit `scale` value to `html2canvas`. Setting `scale: 2` forces the capture at 2× resolution on every device:

```javascript
const canvas = await html2canvas(element, {
  scale: 2,         // force 2× resolution on all devices
  useCORS: true,    // allow cross-origin images (user photo, BJP logo)
  logging: false,
});
```

**Why `scale: 2` specifically?**

- `scale: 1` — low resolution, blurry on all devices
- `scale: 2` — good quality, sharp enough for WhatsApp sharing and A4 printing, low memory cost
- `scale: 3` — very high resolution but uses more memory, risks crashing on 1 GB RAM phones
- `scale: 2` is the safe, high-quality choice for this use case

### What success looks like

- Generate a card on a budget Android phone (or use Chrome DevTools to emulate DPR 1.0: open DevTools → toggle device toolbar → set DPR to 1)
- Download the card
- Open the downloaded image and zoom in to 200%
- Text and edges should be sharp and crisp — not blurry or pixelated
- Generate the same card on a high-end phone
- Both images should look identical in quality when compared side by side

### Does this fix break any existing functionality?

No. `scale: 2` only changes the resolution of the captured image. The card layout, content, download behaviour, and save logic are all unchanged.

One minor note: the output image file size will increase by roughly 4× compared to `scale: 1` (because 2× resolution = 4× the pixels). For a typical member card, this means going from ~150 KB to ~500–600 KB. This is perfectly acceptable for WhatsApp sharing and printing, and has no impact on the server.

## Both Fixes Together

In the final implementation, both fixes should be applied in the same place:

```javascript
// 1. Wait for fonts to load
await document.fonts.ready;

// 2. Capture at consistent resolution
const canvas = await html2canvas(element, {
  scale: 2,
  useCORS: true,
  logging: false,
});

// rest of download logic continues as before...
```
